## gitdb provides the following functionality

- GitHub token is provided via "--github-token" option at command execution time. If not provided, it is read from the environment variable "GITDB_GITHUB_TOKEN". If this is empty, it will be used without a token.
- Repository or pull request can be full text search or semantic search (vector search) with LanceDB.
- Can register GitHub URLs to sync issues and pull requests from repositories.

## Main dependencies

- octocrab: GitHub API client
- lancedb: Embedded vector database for storing repository data and enabling full-text and semantic search
- dirs: Platform-specific standard directories for config, cache, and data paths

### MCP & Client

#### As MCP server

Defined in /bin/gitdb_mcp.rs and under /tools directory

### As client command

The binary /bin/gitdb_cli.rs is executed.

## Database specifications (LanceDB)

### Database schema

The system uses LanceDB as an embedded vector database to store repository metadata, commit history, and search results with support for both full-text and semantic search capabilities.

#### Tables

1. **repositories**
   - id: INTEGER PRIMARY KEY
   - owner: VARCHAR
   - name: VARCHAR
   - full_name: VARCHAR
   - description: TEXT
   - stars: INTEGER
   - forks: INTEGER
   - language: VARCHAR
   - created_at: TIMESTAMP
   - updated_at: TIMESTAMP
   - indexed_at: TIMESTAMP

2. **commits**
   - sha: VARCHAR PRIMARY KEY
   - repository_id: INTEGER REFERENCES repositories(id)
   - author_name: VARCHAR
   - author_email: VARCHAR
   - message: TEXT
   - timestamp: TIMESTAMP
   - parent_sha: VARCHAR

3. **files**
   - id: INTEGER PRIMARY KEY
   - repository_id: INTEGER REFERENCES repositories(id)
   - path: VARCHAR
   - content: TEXT
   - size: INTEGER
   - last_modified: TIMESTAMP

4. **search_cache**
   - id: INTEGER PRIMARY KEY
   - query: VARCHAR
   - search_type: VARCHAR
   - results: JSON
   - created_at: TIMESTAMP
   - expires_at: TIMESTAMP

### Database operations

- Connection management: Automatic connection pooling with configurable limits
- Query optimization: Uses LanceDB's native vector indexing for fast similarity search
- Data persistence: Database and repository data stored in platform-specific data directory:
  - Linux: `~/.local/share/gitdb` or `$XDG_DATA_HOME/gitdb`
  - macOS: `~/Library/Application Support/gitdb`
  - Windows: `%APPDATA%\gitdb`
- Vector embeddings: Automatic generation of embeddings for repository descriptions, issues, and pull requests
- Hybrid search: Supports both keyword-based and semantic similarity search

### Data Storage Locations

gitdb follows the XDG Base Directory Specification and platform conventions to separate configuration and data:

#### Configuration Storage (Immutable)
Application configuration files are stored in platform-specific configuration directories:
- Linux: `~/.config/gitdb` or `$XDG_CONFIG_HOME/gitdb`
- macOS: `~/Library/Preferences/gitdb`
- Windows: `%APPDATA%\gitdb\config`

Configuration includes:
- `config.toml`: Application settings (default search options, API rate limits, etc.)
- `themes.toml`: UI/output formatting preferences
- `aliases.toml`: Command aliases and shortcuts

#### Data Storage (Mutable)
Runtime data, databases, and synchronized content are stored in platform-specific data directories:
- Linux: `~/.local/share/gitdb` or `$XDG_DATA_HOME/gitdb`
- macOS: `~/Library/Application Support/gitdb`
- Windows: `%LOCALAPPDATA%\gitdb`

Data includes:
- `gitdb.lance`: Main LanceDB database containing repositories, issues, PRs, and comments
- `repositories.json`: List of registered GitHub repositories to sync
- `embeddings/`: Cached vector embeddings for semantic search
- `sync_cache/`: Temporary files and ETags for incremental sync optimization

### Performance considerations

- Index on repositories(full_name) for fast repository lookups
- Index on commits(repository_id, timestamp) for efficient commit history queries
- Index on files(repository_id, path) for quick file searches
- Automatic VACUUM scheduled after bulk operations

## Sync Command

### Overview

The `sync` command downloads issues and pull requests from registered repositories, including all their comments. It tracks the last synchronization time and only fetches items that have been modified since the last sync to minimize API calls and improve performance.

### Additional Database Tables

5. **issues**
   - id: INTEGER PRIMARY KEY
   - repository_id: INTEGER REFERENCES repositories(id)
   - number: INTEGER
   - title: VARCHAR
   - body: TEXT
   - state: VARCHAR (open/closed)
   - author: VARCHAR
   - assignees: JSON
   - labels: JSON
   - created_at: TIMESTAMP
   - updated_at: TIMESTAMP
   - closed_at: TIMESTAMP
   - comments_count: INTEGER

6. **pull_requests**
   - id: INTEGER PRIMARY KEY
   - repository_id: INTEGER REFERENCES repositories(id)
   - number: INTEGER
   - title: VARCHAR
   - body: TEXT
   - state: VARCHAR (open/closed/merged)
   - author: VARCHAR
   - assignees: JSON
   - labels: JSON
   - head_ref: VARCHAR
   - base_ref: VARCHAR
   - created_at: TIMESTAMP
   - updated_at: TIMESTAMP
   - merged_at: TIMESTAMP
   - closed_at: TIMESTAMP
   - comments_count: INTEGER
   - commits_count: INTEGER
   - additions: INTEGER
   - deletions: INTEGER
   - changed_files: INTEGER

7. **issue_comments**
   - id: INTEGER PRIMARY KEY
   - issue_id: INTEGER REFERENCES issues(id)
   - comment_id: INTEGER
   - author: VARCHAR
   - body: TEXT
   - created_at: TIMESTAMP
   - updated_at: TIMESTAMP

8. **pull_request_comments**
   - id: INTEGER PRIMARY KEY
   - pull_request_id: INTEGER REFERENCES pull_requests(id)
   - comment_id: INTEGER
   - author: VARCHAR
   - body: TEXT
   - created_at: TIMESTAMP
   - updated_at: TIMESTAMP

9. **sync_status**
   - id: INTEGER PRIMARY KEY
   - repository_id: INTEGER REFERENCES repositories(id)
   - resource_type: VARCHAR (issues/pull_requests)
   - last_synced_at: TIMESTAMP
   - status: VARCHAR (success/failed/in_progress)
   - error_message: TEXT
   - items_synced: INTEGER

### Command Usage

```bash
# Sync all registered repositories
gitdb sync

# Sync specific repository
gitdb sync --repo owner/name

# Force full sync (ignore last sync time)
gitdb sync --full

# Sync all issues and PRs without considering latest synced date
gitdb sync all

# Sync only issues or pull requests
gitdb sync --type issues
gitdb sync --type pull-requests
```

### Sync Logic

1. **Check last sync time**: Query `sync_status` table for each repository (skipped when using `sync all`)
2. **Fetch updates**: Use GitHub API with `since` parameter set to last sync time (or fetch all when using `sync all`)
3. **Incremental update**: Only fetch and update items modified after last sync (or all items when using `sync all`)
4. **Sync comments**: For each issue/PR, also sync all associated comments
5. **Transaction handling**: All updates wrapped in database transaction
6. **Error recovery**: Failed syncs don't update last_synced_at, allowing retry

#### Sync All Behavior

When using `gitdb sync all`:
- Ignores the `last_synced_at` timestamp in the `sync_status` table
- Fetches all issues and pull requests from the beginning of the repository
- Replaces existing data with fresh data from GitHub
- Still respects rate limits and implements proper error handling
- Useful for initial sync or when data integrity needs to be restored

### API Optimization

- Uses conditional requests with ETags when available
- Implements rate limit handling with automatic retry
- Batches database operations for better performance
- Parallel sync for multiple repositories (configurable concurrency)

### Indexes for Sync Operations

- Index on issues(repository_id, updated_at)
- Index on pull_requests(repository_id, updated_at)
- Index on sync_status(repository_id, resource_type)
- Unique index on issues(repository_id, number)
- Unique index on pull_requests(repository_id, number)
- Index on issue_comments(issue_id, created_at)
- Index on pull_request_comments(pull_request_id, created_at)
